# ライセンシー 事前準備チェックリスト

**対象:** このプラットフォームをライセンス契約で利用する事業者（以下「ライセンシー」）

**このドキュメントについて:**
このプラットフォームを自分のブランドで運営するために必要な「全ての準備」をまとめたチェックリストです。技術的な経験がゼロでも理解できるよう、各項目の「なぜ必要か」「どこで使われるか」「個人名義でいいのかビジネス名義が必要なのか」を詳しく説明しています。

**重要:**
- セットアップウィザードを開始する **前に** このリストを全て準備しておくと、作業がスムーズに進みます
- 途中でアカウント登録に詰まると、全体の作業が止まります
- まずこの全リストを読んで、何が必要かを把握してから作業を始めてください

**所要時間の目安:**
| 作業 | 目安 |
|------|------|
| ブランド素材の準備（デザイナーへの依頼含む） | 数日〜2週間（事前に着手を） |
| 各種アカウント作成と設定 | 60〜90分 |
| API キーの取得・メモ | 30〜60分 |
| Vercel デプロイ＋環境変数設定 | 20〜30分 |
| セットアップウィザード完了 | 10〜15分 |

### 1-3. ロゴファイル（推奨・任意）

**何のためか:** 全ページのヘッダー左上に表示され、プラットフォームの第一印象を決める最重要素材です。注文確認メールにも表示されます。

**準備すること:**
- デザイナーに依頼するか、Canva等のツールで作成

## 🖥️ Section 2: 必要なアカウント一覧と取得手順

### 📌 アカウント登録に使うメールアドレス・アカウントについて

**結論：事業者用（ビジネス用）のGoogleアカウントまたはメールアドレスで全サービスを統一することを強く推奨します。**

理由は3つあります：

1. **管理の一元化**: 全サービスが同じビジネスアカウントに紐づくため、将来スタッフに引き継いだり、会社として管理する際に整理されています
2. **信頼性・プロフェッショナリズム**: 各サービスのダッシュボードに表示される名前・メールが「ビジネス名義」になることで、サービス事業者としての信頼性が高まります
3. **アカウント喪失リスクの低減**: 担当者の個人アカウントで登録すると、その人が退職した際に全サービスへのアクセスを失うリスクがあります

**具体的に:**
- 既にビジネス用のGoogleアカウント（Google Workspace）をお持ちの場合は、そのアカウントで全サービスに登録するのが理想です
- Apple アカウント（Apple Business Manager）もある場合、Apple でのサインアップに対応しているサービスではそちらを使っても良いですが、Google のほうが対応サービスが多いため Google を主軸にすることを推奨します
- 例外: Printful はビジネスアカウントという区分がなく、登録したメールアドレスがビジネス用であれば十分です

### 2-3. Turso（データベース）

**理想の登録方法:** ビジネス用メールアドレスで登録します。スタートアップ段階でも「会社として使っているDB」として管理を統一しておくことで、スタッフへの引き継ぎや監査が容易になります。

**プランの選択:**

   - **Database URL**: `libsql://データベース名-ユーザー名.turso.io` の形式

### 2-4. Cloudflare（ファイルストレージ R2）

**何のためか:** 利用者がアップロードしたロゴファイルやデザインファイルを保存する「クラウドストレージ」です。Google Drive のような場所ですが、プログラムから自動的に保存・取り出しができます。

**理想の登録方法:** ビジネス用メールアドレス（またはビジネス用Googleアカウント）で登録します。Cloudflare はインフラの根幹（ドメイン・DNS・ストレージ）を担うサービスであり、会社名義で管理しておくことが特に重要です。将来、独自ドメインのDNSを Cloudflare で管理することにすれば、一元管理できて便利です。

**プランの選択:**
- **Free（無料）**: R2 は 10GB/月まで無料で、ほとんどのサービス規模で十分です
- ドメインを Cloudflare に移管した場合も、DNS管理は無料です

**登録・設定手順:**

**Part 1: バケット作成 + Public URL 取得**

1. [cloudflare.com](https://cloudflare.com) にアクセス → 「Sign Up（Free）」
2. ビジネス用メールアドレスでアカウント作成
3. ログイン後、左メニューまたは検索で **R2 Object Storage** を開く
4. 「Create bucket」をクリック
   - バケット名を入力（例: `schoolmerch-uploads`）
   - Location: Automatic（変更不要）
   - 「Create bucket」をクリック
5. 作成されたバケット名をクリック → **Settings** タブを開く
6. **Public access** セクション → 「Allow Access」→ 確認ダイアログで承認
7. 表示された **Public development URL**（例: `https://pub-xxxxx.r2.dev`）をメモ

**Part 2: Account ID の確認**

8. ブラウザの URL バーを見ると `https://dash.cloudflare.com/{32文字の英数字}/...` という形式になっています
9. この `{32文字の英数字}` 部分が **Account ID** です。コピーしてメモしてください

**Part 3: R2 API トークンの生成（S3互換認証情報）**

10. R2 のトップページを開き、右上または左ナビの「**Manage R2 API Tokens**」をクリック
11. 「**Create Account API token**」をクリック（Recommended と表示されているほう）
12. フォームを入力:
    - **Token name**: 任意（例: `schoolmerch-uploads-rw`）
    - **Permissions**: 「**Object Read & Write**」を選択（ファイルのアップロード・表示に必要）
    - **Specify bucket(s)**: 「**Apply to all buckets in this account**」を選択
    - TTL や IP フィルタリングは設定不要（デフォルトのまま）
13. 「**Create Account API Token**」ボタンをクリック
14. 完了画面に3種類の認証情報が表示されます。**S3 クライアント用（中段）** のみ使用します:

    ```
    ✅ S3 clients 用（こちらを使う）:
    Access Key ID:      xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx（32文字）
    Secret Access Key:  xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx（64文字）

    ❌ Token value（cfat_...）は使わない:
    これは Cloudflare API 直接呼び出し用。S3 互換 API（r2.ts）では不要。
    ```

15. **「この画面を閉じると二度と表示されない」** — 必ず今すぐメモしてください

| 準備状況 | Account ID | Bucket名 | Access Key ID | Secret Access Key | Public URL |
|---------|-----------|---------|-------------|-----------------|-----------|
| ☐ | URL から取得 | &nbsp; | **必ずメモ！** | **必ずメモ！** | `https://pub-xxx.r2.dev` |

### 2-5. Stripe（決済）

**何のためか:** 購入者がクレジットカードで商品代金を支払うための決済サービスです。また「Stripe Connect」という機能で、各組織（PTA等）の銀行口座に売上を自動で振り込む機能も担います。

**理想の登録方法:** ビジネス用メールアドレスで登録し、**法人または個人事業主として**アカウントを設定することを最初から強く推奨します。テスト段階でも最初から会社名義で登録しておくと、本番移行時の再設定が不要になります。

- **テスト段階**: テストモード（`sk_test_...`）では本人確認なしで使えますが、アカウント自体はビジネス名義で作成しておく
- **本番公開（実際にお金を動かす）**: Stripe は本人確認（KYC）を要求します
  - 個人事業主の場合: 個人名・住所・生年月日・マイナンバー等
  - 法人の場合: 法人登録情報・代表者情報等
  - ⚠️ **審査には数日かかることがあります。本番公開予定日の2週間前までに着手することを推奨します**

**テストモードと本番モードの切り替え:** Stripe はテストモード（`sk_test_...`）と本番モード（`sk_live_...`）が分かれています。まずテストモードで全機能を確認してから、本番モードに切り替えてください。

**登録・設定手順:**
1. [stripe.com](https://stripe.com) にアクセス → 「アカウントを作成」
2. メールアドレス・パスワードを設定
3. ダッシュボード右上の「テストモード」スイッチが ON になっていることを確認
4. 左メニュー → **Developers** → **API keys** から以下をメモ:
   - **Publishable key**: `pk_test_...`（公開可能なキー）
   - **Secret key**: `sk_test_...`（秘密キー — 絶対に他人に教えない）
5. Webhook Secret は後でも取得可能（セットアップ後に Stripe CLI で取得する方法を推奨 — 詳細は system-design.md 参照）

| 準備状況 | Publishable Key | Secret Key | Webhook Secret |
|---------|--------------|-----------|--------------|
| ☐ | `pk_test_...` | `sk_test_...` | 後で設定可 |

### 2-6. Printful（印刷・発送）

**何のためか:** 注文が入ったTシャツやその他グッズを印刷して購入者の自宅に発送する「プリント・オン・デマンド（POD）」サービスです。在庫を持たずに販売できる仕組みの核心部分です。

**理想の登録方法:** Printful にはビジネス・個人の区別がなく、登録するメールアドレスに差はありません。ただし、**ビジネス用メールアドレスで統一して登録する**ことを推奨します。これはアカウント管理の一貫性と、将来のスタッフ引き継ぎのためです。

**重要:** Printful のアカウントはプラットフォームオーナー（ライセンシー）が持つものです。購入者や Org の管理者が Printful アカウントを持つ必要はありません。Printful への注文送信はプラットフォームが自動で行います。

**登録・設定手順:**
1. [printful.com](https://printful.com) にアクセス → 「Get started for free」
2. メールアドレスで登録（Google や Apple でのソーシャルログインも可）
3. ダッシュボードから **Stores** → 「Connect via API」をクリック
4. モーダルが開いたら:
   - Store name を入力（例: `Community Merch Platform`）
   - 「Continue」をクリック
5. ストアが作成されたら、Settings → Store settings → **API** → 「Go to Printful Developers」をクリック
6. 新しいタブで **Printful Developers** サイトが開く → 「Sign in to Printful」
7. ログイン後、左メニュー **Tokens** → 右上「+」ボタン（Add new token）
8. フォームに入力:
   - **Token name**: 任意（例: `Community Merch Platform Dev`）
   - **Contact email**: 自分のメールアドレス
   - **Expiration date**: 最大 2 年先の日付（カレンダーから選択 — 必須項目）
   - **Access level**: "A single store" を選択 → 先ほど作ったストアを選択
   - **Scopes**: 「View and manage orders of the authorized store」にチェック
9. 「Create new token」をクリック
10. 表示された API Token を**必ずメモ**（この画面を閉じると二度と表示されません）

| 準備状況 | API Token | ストア名 | 支払い方法の登録 |
|---------|---------|--------|---------------|
| ☐ | **必ずメモ！** | &nbsp; | ☐ **必須**（Billing → Payment methods にカード登録。未登録だと注文の自動確定が失敗し発送されません） |

### 2-7. Resend（メール送信）

**何のためか:** 注文確認メール・マジックリンク（メールアドレスでのサインイン）・発送通知メールを送信するサービスです。

**理想の登録方法:** ビジネス用メールアドレスで登録し、**独自ドメインのメールアドレスをすぐに設定する**ことを強く推奨します（例: `noreply@schoolmerch.org`）。メールの「差出人」が自社ドメインになることで、利用者からの信頼感が大きく向上し、迷惑メールフォルダに入りにくくなります。

**プランの選択:**
- **Free（無料）**: 100通/日。テスト・小規模には十分
- **Pro（月 $20 〜）**: 本番運用では Pro を推奨。送信数制限なし・優先サポートあり

**重要 — ドメイン認証について:**
- 設定しない場合: メールは `resend.dev` ドメインから送信されます（迷惑メール扱いされる可能性あり）
- 設定する場合: 独自ドメインのメールアドレス（例: `noreply@schoolmerch.org`）から送信でき、信頼性が大幅に向上します
- ドメイン認証には独自ドメインの DNS 設定が必要です（ドメインを持っている場合のみ可能）

**登録・設定手順:**
1. [resend.com](https://resend.com) にアクセス → 「Start for free」
2. メールアドレスで登録
3. ダッシュボード左メニュー → **API Keys** → 「Create API Key」
4. 任意の名前を入力 → 「Add」
5. 生成された API Key をメモ（`re_...` で始まる文字列）
6. （推奨）独自ドメインがある場合: **Domains** → 「Add Domain」→ DNS レコードを追加してドメイン認証

| 準備状況 | API Key | 送信元メールアドレス | ドメイン認証 |
|---------|--------|------------------|-----------|
| ☐ | `re_...` | 例: `noreply@yourdomain.com` | ☐ 完了 |

### 2-8. Google Cloud Console（Google ログイン機能）

**何のためか:** 利用者がGoogleアカウントでサインインできる「Googleでログイン」機能のために必要な設定です。これを設定しない場合、メールアドレス（マジックリンク）のみでのサインインになります。

**理想の登録方法:** **ビジネス用 Google アカウント（Google Workspace）で設定することを強く推奨します。**

理由：Google Cloud Console で作成するOAuth アプリの「同意画面」には登録したアカウントの組織名が表示されます。個人アカウントで作成すると「未確認のアプリ」として警告が出ることがあり、利用者に不安を与えます。ビジネスアカウントで作成し、「確認済みドメイン」を設定することで、利用者が安心してGoogleログインを使えるようになります。

**登録・設定手順:**
1. [console.cloud.google.com](https://console.cloud.google.com) にアクセス（Googleアカウントでサインイン）
2. 右上の「プロジェクト」から「新しいプロジェクト」を作成（例: `SchoolMerch Auth`）
3. 左メニュー → **APIs & Services** → **Credentials**
4. 「Create Credentials」→「OAuth client ID」
5. Application type: **Web application**
6. Name: 任意（例: `SchoolMerch Web`）
7. **Authorized redirect URIs** に追加:
   - 開発用: `http://localhost:3000/api/auth/callback/google`
   - 本番用: `https://yourdomain.com/api/auth/callback/google`（デプロイ後に追加）
8. 「Create」→ **Client ID** と **Client Secret** をメモ
9. OAuth 同意画面の設定も必要（左メニュー → **OAuth consent screen** → External → 基本情報を入力）

| 準備状況 | Client ID | Client Secret |
|---------|---------|-------------|
| ☐ | `xxx.apps.googleusercontent.com` | メモ済み |

### 2-9. OpenAI（AI デザイン生成 — オプション）

**何のためか:** キャンペーン作成時にAIがデザイン案を自動生成する機能のために必要です。設定しない場合はこの機能が使えませんが、**プラットフォーム全体の動作には影響しません**。

**理想の登録方法:** ビジネス用メールアドレスで登録し、**会社のクレジットカードで支払い設定をする**ことを推奨します。AI 生成の利用料は使用量に応じて増えるため、個人クレジットカードと混在しない方が経費管理がしやすいです。

**費用:** 画像生成 1枚あたり約 $0.04（約6円）。クレジットカードの登録が必要です。

**登録手順:**
1. [platform.openai.com](https://platform.openai.com) にアクセス → 「Sign Up」
2. メールアドレスで登録（または Google・Apple でサインアップ）
3. 支払い情報を設定（Settings → Billing → Add payment method）
4. 左メニュー → **API Keys** → 「Create new secret key」
5. 生成されたキー（`sk-...`）をメモ（この画面を閉じると再表示されません）

| 準備状況 | API Key |
|---------|--------|
| ☐ (任意) | `sk-...` |

## 🔑 Section 3: 環境変数まとめ（コピペ用テンプレート）

Vercel のデプロイ設定「Environment Variables」に入力する値の完全リストです。以下の空欄に取得した値を記入しておくと作業がスムーズです。

```bash
# ─── Turso Database ───────────────────────────────────────────────────────
TURSO_DATABASE_URL=libsql://ここに入力.turso.io
TURSO_AUTH_TOKEN=ここに入力

# ─── Better Auth ─────────────────────────────────────────────────────────
# 以下のコマンドで生成: openssl rand -base64 32
BETTER_AUTH_SECRET=ここに入力（32文字以上のランダム文字列）
BETTER_AUTH_URL=https://あなたのドメイン.com
NEXT_PUBLIC_APP_URL=https://あなたのドメイン.com

# ─── Google OAuth ─────────────────────────────────────────────────────────
GOOGLE_CLIENT_ID=ここに入力.apps.googleusercontent.com
GOOGLE_CLIENT_SECRET=ここに入力

# ─── Resend（メール送信）──────────────────────────────────────────────────
RESEND_API_KEY=re_ここに入力
EMAIL_FROM=noreply@あなたのドメイン.com

# ─── Stripe（決済）───────────────────────────────────────────────────────
STRIPE_SECRET_KEY=sk_test_ここに入力（本番は sk_live_...）
STRIPE_PUBLISHABLE_KEY=pk_test_ここに入力（本番は pk_live_...）
STRIPE_WEBHOOK_SECRET=whsec_ここに入力
# 団体の口座連携の完了検知用（Connected accounts エンドポイントのシークレット・別途発行）
STRIPE_CONNECT_WEBHOOK_SECRET=whsec_ここに入力

# ─── Cloudflare R2（ファイルストレージ）────────────────────────────────────
CLOUDFLARE_R2_ACCOUNT_ID=ここに入力
CLOUDFLARE_R2_ACCESS_KEY_ID=ここに入力
CLOUDFLARE_R2_SECRET_ACCESS_KEY=ここに入力
CLOUDFLARE_R2_BUCKET_NAME=ここに入力（例: schoolmerch-uploads）
CLOUDFLARE_R2_PUBLIC_URL=https://pub-ここに入力.r2.dev

# ─── Printful（印刷・発送）───────────────────────────────────────────────
PRINTFUL_API_KEY=ここに入力
# 以下のコマンドで生成: openssl rand -hex 20
PRINTFUL_WEBHOOK_SECRET=ここに入力（ランダム文字列）

# ─── Platform Admin 保護 ──────────────────────────────────────────────────
# セットアップウィザードを完了できる人を、あなた自身のメールアドレスに限定します
# 必ず設定してください（設定しないと誰でも管理者になれてしまいます）
PLATFORM_ADMIN_EMAIL=あなたのメールアドレスをここに入力（例: you@example.com）

# ─── Cron Job セキュリティ────────────────────────────────────────────────
# 以下のコマンドで生成: openssl rand -hex 32
# または https://www.uuidgenerator.net/ 等で生成したランダム文字列でもOK
CRON_SECRET=ここに入力（ランダム文字列）

# ─── OpenAI（オプション）────────────────────────────────────────────────
# OPENAI_API_KEY=sk-ここに入力
```

> **`CRON_SECRET` について:**
> これは「毎日自動で実行される裏側の処理（モックアップ画像の更新等）」を守るための合言葉です。第三者がその処理を不正に呼び出せないようにするために必要です。推測されにくいランダムな文字列を設定してください。

> **`openssl` コマンドについて:**
> - Mac/Linux のターミナルで使えます
> - Windows の場合は [UUID Generator](https://www.uuidgenerator.net/) 等のオンラインツールで代替可能
> - `BETTER_AUTH_SECRET`: 任意の32文字以上の文字列でOK（推測されないランダムなものを使用）
> - `PRINTFUL_WEBHOOK_SECRET`: 任意の文字列でOK（Printful からの webhook を認証するためのパスワード）

## 📦 Section 4: デプロイ手順（概要）

1. **プラットフォーム提供者からリポジトリへのアクセスを受け取る**
   - GitHubにコラボレーター招待またはリポジトリのフォーク権限が届きます
   - 招待メールが届いたら、ビジネス用GitHubアカウントでログインして承認してください
   - わからない場合はプラットフォーム提供者にご連絡ください

2. **Vercel にデプロイする**
   - リポジトリページ右上の **「Fork」** をクリックして、自分のアカウントに複製を作る
     - ⚠️ Fork せずにコピーしてデプロイすると、将来のアップデートを受け取れなくなります
   - [vercel.com/new](https://vercel.com/new) を開き、GitHub 連携して Fork したリポジトリを「Import」
   - 環境変数（Section 3 のテンプレートに値を埋めたもの）を Environment Variables 欄に**まとめて貼り付け**
   - 「Deploy」をクリック → 数分で完了
   - 以後のアップデートは、Fork のページで「Sync fork」→「Update branch」を押すだけで自動反映されます

3. **独自ドメインを設定する（任意）**
   - Vercel Dashboard → Domains → ドメインを追加
   - レジストラ（お名前.com等）で DNS レコードを変更

3.5. **プラットフォームに初めてサインインする**

   デプロイが完了したら、あなた自身がプラットフォームにサインインする必要があります。

   **サインイン方法は2種類あります（どちらでもOK）:**

   | 方法 | 使える条件 | 手順 |
   |------|-----------|------|
   | **Googleでサインイン** | Google アカウントを持っている | 「Continue with Google」をクリックするだけ |
   | **メールアドレスでサインイン（マジックリンク）** | メールアドレスがあれば誰でも | メールアドレスを入力 → 届いたメールのリンクをクリック |

   > **マジックリンクとは？**
   > パスワード不要のサインイン方法です。メールアドレスを入力すると「サインイン用リンク」が届き、そのリンクをクリックするだけでサインインできます。Googleアカウントがなくてもご利用いただけます。ただし、メール送信には Resend（Section 2-7）の設定が必要です。Resend が未設定の場合はGoogleサインインをご使用ください。

   **手順:**
   1. `https://あなたのドメイン.vercel.app/sign-in` にアクセス
   2. Googleアカウントがある場合 → 「Continue with Google」をクリック
   3. Googleアカウントがない・使いたくない場合 → メールアドレスを入力して「Send sign-in link」をクリック → 届いたメールのリンクをクリック
   4. ダッシュボードが表示されればサインイン成功です

4. **`/setup` ウィザードを完了する（9ステップ）**

   | ステップ | 内容 | 使う準備物 |
   |---------|------|---------|
   | Step 1 | ライセンス同意 | — |
   | Step 2 | ブランド設定 | Section 1 の全情報 |
   | Step 3 | Database (Turso) | env var 設定確認 |
   | Step 4 | Payments (Stripe) | env var 設定確認 |
   | Step 5 | Printful | env var 設定確認 |
   | Step 6 | Email (Resend) | env var 設定確認 |
   | Step 7 | AI Design（任意） | env var 設定確認 |
   | Step 8 | File Storage (R2) | env var 設定確認 |
   | Step 9 | Review & Launch | — |

5. **「🚀 Launch Platform」をクリック → サービス開始！**

## ⚠️ Section 5: よくある注意事項・落とし穴

### Stripe（決済）
- **テストモードと本番モードを間違えないこと。** `sk_test_...` と `sk_live_...` は別物です。本番に切り替えるまでは絶対に `sk_test_...` を使用してください。
- **本番利用には本人確認（KYC）が必要で、完了まで数日かかります。** 公開日が決まっている場合は2週間前には着手してください。
- **Stripe Connect のオンボーディング（銀行口座連携）は各 Org 管理者が行います。** これはプラットフォーム設定とは別の作業です。

### Printful（印刷・発送）
- **API Token は一度しか表示されません。** 「Create new token」をクリックした直後に必ずコピーして安全な場所に保存してください。
- **Printful のストアを先に作成してからトークンを生成してください。** ストアなしではトークンに適切な権限が付与できません。

### Cloudflare R2（ファイルストレージ）
- **Access Key ID と Secret Access Key は一度しか表示されません。** 生成直後に必ずメモしてください。
- **バケットの Public Access を必ず有効化してください。** 有効化しないとアップロードした画像が表示されません。

### Google OAuth（Googleログイン）
- **リダイレクト URI は正確に設定してください。** `https://` と `http://` の違い、末尾のスラッシュの有無も区別されます。
- **本番ドメインを取得後、リダイレクト URI を追加忘れずに。** 忘れるとGoogleログインが本番で動きません。

### Resend（メール）
- **独自ドメインのメールを使う場合、ドメイン認証をしないとスパム扱いされる可能性があります。**
- **DNS の変更が反映されるまで最大 48 時間かかることがあります。** ドメイン認証は早めに行ってください。

### 環境変数全般
- **秘密キー（Secret Key, API Token 等）は絶対に公開しないでください。** GitHubにアップロードしたり、メールで送ったりしないこと。メモする場合は 1Password や Bitwarden 等のパスワードマネージャーの使用を強く推奨します。
- **`BETTER_AUTH_URL` と `NEXT_PUBLIC_APP_URL` は本番の HTTPS URL にしてください。** `http://localhost:3000` のままだと本番でサインインが動きません。
- **`CRON_SECRET` は必ず設定してください。** 設定しないと、毎日の自動処理（モックアップ画像の更新等）が正常に動作しません。`openssl rand -hex 32`（Mac/Linux）または オンラインのランダム文字列生成ツールで生成してください。
